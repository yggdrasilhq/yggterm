# Demo Style

The demo system should produce assets that feel calm, technical, and trustworthy.

## Tone

- evidence-first
- restrained
- legible at a glance
- specific about what changed

Avoid:

- noisy transitions
- vague marketing copy
- effects that distract from the UI itself

## Motion

Use motion to explain state changes:

- zoom to the relevant region
- highlight a row, chip, or viewport
- reveal before/after differences
- pace transitions so a technical audience can follow what changed

The visual target is closer to a careful explanatory clip than a flashy trailer.

## Annotation

Annotations should be:

- short
- literal
- tied to visible evidence

Examples:

- `Preview stays stable after 23s dwell`
- `Machine row turns green after remote refresh`
- `Rendered graph mode uses conversation nodes instead of summary cards`

## Composition

For screenshots and videos:

- crop to the relevant part of the app when possible
- keep context when the surrounding chrome matters
- avoid clutter from unrelated desktops or terminals unless needed for debugging proof

## Release Pages

Release pages should use:

- one strong screenshot per feature
- optional short clips for motion-heavy fixes
- proof summaries that read like product wins, not commit logs
